# 三端店铺详情与预约总结金额时长合计

日期：2026-03-23

## 本次目标

统一 iOS、Android、H5 三端在店铺详情页和预约流程中的金额、时长展示逻辑：

- 店铺详情页底部 `Continue` 区域
- 预约弹窗/预约页面顶部服务摘要
- `Appointment Summary` 预约总结卡
- 成功态中的总金额/总时长展示

要求：

- 选中多个服务时，金额按已选服务价格求和
- 时长按已选服务 duration 求和
- 相关联的时段/冲突判断也要同步使用合计时长，避免 UI 和实际逻辑不一致

### iOS

- `ios-app/NailsDashIOS/Sources/Features/Appointments/BookAppointmentView.swift`

## 具体修复

### 1. 店铺详情页底部 Continue 区域

iOS、Android 都从“取第一个服务的 price / duration”改成“对已选服务列表求和”：

- 总金额：`sum(price)`
- 总时长：`sum(duration_minutes)`

### 2. 预约弹窗顶部服务摘要

iOS、Android 的预约弹窗顶部摘要改为：

- 金额显示合计总价
- 时长显示合计总时长
- 服务标题在多服务时显示为 `首个服务名 +N`

### 3. Appointment Summary 预约总结卡

iOS、Android、H5 的预约总结区统一改为：

- 金额显示合计总价
- Duration 显示合计总时长
- Service 行在多服务时显示聚合标题，而不是只显示首个服务

### 4. H5 本地时长逻辑同步修复

H5 原来底部展示已经是合计，但以下逻辑还在按第一个服务计算：

- 本地预约冲突判断
- 基于店铺营业时间生成可预约时段
- 按 blocked slots 过滤时间
- 预约成功回传的 `totalPrice / totalDuration`

本次已同步改为：

- 单人多服务：按已选服务合计时长计算
- 团单：仍按 host service + guest services 各自逻辑处理

## 当前边界

前端三端的展示和 H5 本地时间计算已经统一。

但 iOS / Android / H5 创建预约时，后端主创建接口仍然只接收一个主 `service_id`。因此：

- 本次已修正三端展示汇总
- H5 已修正本地可预约时段和冲突判断
- 如果后续需要让预约记录的服务项、订单金额、后端时段逻辑也完全以多服务为准，需要继续把客户端接到后端的 `appointment service items` 接口
