# 开发更新 2026-03-23：三端预约多服务落库与后台展示对齐

## 背景
- 之前三端店铺详情页和预约总结已经按多服务合计金额、时长展示。
- 但预约创建主流程仍主要依赖 `appointments.service_id`，导致多服务只停留在前端展示层。
- 后台管理系统 `Appointments` 列表、Timeline、详情页，以及 `Dashboard` 底部实时通知，也仍然只展示主服务名。

## 本次改动

### 后端
- 放宽 `appointment service items` 接口权限，允许预约所有者在未取消、未结算前维护自己的服务项。
- 单独补充管理端列表和 dashboard 所需的多服务聚合逻辑：
  - `GET /appointments/admin` 返回聚合后的：
    - `service_name`
    - `service_duration`
    - `order_amount`
    - `service_items`
  - `GET /dashboard/realtime-notifications` 返回：
    - `service_name`
    - `service_items`
- 复用 `appointment_service_items` 与 `services` 进行批量汇总，避免前端逐行二次请求。

### Android / iOS / H5
- 单人预约创建后：
  - 先创建主预约
  - 再将主服务之外的附加服务写入 `appointment service items`
- 预约列表 / 预约详情 / 预约总结，统一优先读取 `service_items` 汇总信息。
- 金额与时长显示改为来自真实服务项汇总，不再只依赖主服务。

### Admin Dashboard
- `Appointments` 页面：
  - 列表服务列支持多服务逐行显示
  - Timeline 卡片支持多服务逐行显示
  - 独立 `Appointment Detail` 页支持多服务逐行显示
  - 服务筛选逻辑兼容多服务
- `Dashboard` 页面：
  - 底部实时通知支持多服务逐行显示
  - 通知接口与前端类型同步增加 `service_items`

## 影响范围
- 预约创建
- 预约列表
- 预约详情
- 预约总结
- 后台管理系统 `Appointments`
- 后台管理系统 `Dashboard`

- Android：`cd android-app && ./gradlew :app:assembleDebug -x lint`

- H5：`cd frontend && npm run build`
