# 开发更新 2026-03-20: Backend 支付链路真实回归清单

## 目标

整理一份可重复执行的后端真实回归清单，覆盖以下核心支付链路：

1. 用户注册与登录
2. 优惠券创建与领取
3. 礼品卡购买
4. 预约创建与确认
5. `settle` 结算
6. `refund` 部分退款
7. `refund` 全额退款
8. 结算后积分、优惠券、礼品卡余额回滚校验

本清单基于本地真实运行环境执行，不使用测试替身：

- `backend`
- `backend-scheduler`
- `db`
- `redis`

## 前置条件

1. `docker compose up --build -d`
2. `backend`、`backend-scheduler`、`db`、`redis` 全部 `Up`
3. API 可访问：`http://localhost:8000/api/docs`
4. 数据库已完成 schema 初始化
5. 回归前先清空本轮业务测试数据，避免旧数据污染主键和余额状态

## 最小种子数据

以下数据不走前台 API，直接作为后台初始化种子插入：

1. 1 个超级管理员
2. 1 个店铺
3. 1 个服务
4. 7 天营业时间

推荐基线：

- 店铺：`Regression Payment Salon`
- 服务：`Structured Gel Set`
- 服务价格：`70.0`
- 营业时间：`09:00-19:00`
- 超管登录手机号：`2125550198`

说明：

- 超级管理员用于调用 `/coupons/create`、`/appointments/{id}/settle`、`/appointments/{id}/refund`
- 顾客用户必须走真实注册接口创建

## 回归步骤

### A. 顾客注册与登录

1. `POST /api/v1/auth/send-verification-code`
2. `POST /api/v1/auth/verify-code`
3. `POST /api/v1/auth/register`
4. `POST /api/v1/auth/login`

校验点：

- 注册成功返回 `201`
- 顾客 `access_token` 可正常访问受保护接口

### B. 管理员登录

1. `POST /api/v1/auth/login`

请求要点：

- `login_portal = admin`

校验点：

- 超级管理员可以成功拿到 `access_token`

### C. 优惠券链路

1. 超管 `POST /api/v1/coupons/create`
2. 顾客 `POST /api/v1/coupons/claim`
3. 顾客 `GET /api/v1/coupons/my-coupons`

推荐用例：

- 固定金额券
- `discount_value = 10`
- `category = activity`
- `valid_days = 30`

校验点：

- `user_coupon.status = available`

### D. 礼品卡链路

1. 顾客 `POST /api/v1/gift-cards/purchase`
2. 顾客 `GET /api/v1/gift-cards/my-cards`
3. 顾客 `GET /api/v1/gift-cards/summary`

推荐用例：

- 顾客自购礼品卡 `25`

校验点：

- 礼品卡状态 `active`
- 礼品卡余额 `25`

### E. 预约链路

1. 顾客 `POST /api/v1/appointments/`
2. 超管 `PATCH /api/v1/appointments/{id}/confirm`

推荐用例：

- 预约时间取次日 `10:00`
- 避免过去时间和非营业时间

校验点：

- 预约创建成功
- 预约状态变为 `confirmed`

### F. 结算链路

1. 超管 `POST /api/v1/appointments/{id}/settle`

推荐结算组合：

- `original_amount = 70`
- `user_coupon_id = 已领取优惠券`
- `coupon_discount_amount = 10`
- `gift_card_id = 顾客礼品卡`
- `gift_card_amount = 25`
- `cash_paid_amount = 35`
- `idempotency_key = settle-001`

预期结果：

- `settlement_status = settled`
- `coupon_discount_amount = 10`
- `gift_card_used_amount = 25`
- `cash_paid_amount = 35`
- `final_paid_amount = 60`

附加校验：

1. 顾客积分余额变为 `60`
2. 顾客优惠券状态变为 `used`
3. 顾客礼品卡余额变为 `0`

### G. 部分退款链路

1. 超管 `POST /api/v1/appointments/{id}/refund`

推荐退款组合：

- `refund_cash_amount = 10`
- `refund_gift_card_amount = 5`
- `gift_card_id = 同一张礼品卡`
- `idempotency_key = refund-001`

预期结果：

- `settlement_status = partially_refunded`
- `cash_paid_amount = 25`
- `gift_card_used_amount = 20`
- `final_paid_amount = 45`

附加校验：

1. 顾客积分余额变为 `45`
2. 顾客礼品卡余额变为 `5`
3. 顾客优惠券仍为 `used`

### H. 全额退款收口链路

1. 再次调用 `POST /api/v1/appointments/{id}/refund`

推荐退款组合：

- `refund_cash_amount = 25`
- `refund_gift_card_amount = 20`
- `gift_card_id = 同一张礼品卡`
- `idempotency_key = refund-002`

预期结果：

- `settlement_status = refunded`
- `cash_paid_amount = 0`
- `gift_card_used_amount = 0`
- `final_paid_amount = 0`

附加校验：

1. 顾客积分余额回到 `0`
2. 顾客礼品卡余额回到 `25`
3. 顾客优惠券状态回到 `available`
4. 顾客预约列表中的 `settlement_status = refunded`
