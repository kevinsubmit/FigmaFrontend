# 2026-03-22 Backend 统一数据源与媒体引用修复

## 背景
- 本机同时存在两套后端：
  - `launchctl` 常驻的 `uvicorn + SQLite`
  - `docker compose` 的 `backend + MySQL`
- 结果是：App 端和后台管理端读到的不是同一套数据。

## 本次处理
1. 停用并永久禁用 `com.nailsdash.backend` LaunchAgent，避免 SQLite 后端再次抢占 `8000`。
2. 将 `backend/nailsdash.db` 中的业务数据迁移到 Docker MySQL `nailsdash`。
3. 将旧 `backend/uploads` 同步到 Docker `backend` 的 `/app/uploads`。
4. 清空 Redis 缓存并重启 `backend` / `backend-scheduler`。
5. 新增媒体引用修复脚本：
   - `backend/repair_upload_references.py`
6. 清理两类脏数据：
   - 指向不存在文件的 `/uploads/...` 引用
   - 非法 scheme 的媒体 URL（例如 `javascript:alert(1)`）

## 迁移后关键数据
- `backend_users = 12`
- `stores = 6`
- `services = 20`
- `pins = 11`
- `promotions = 4`
- `appointments = 104`
- `notifications = 149`

## 具体修复结果
- 已删除 3 条已标记删除、但仍指向失效 `/uploads` 图片的测试 pin。
- 已将 1 条非法头像 URL 置空：
  - `backend_users.id = 4`
  - 原值：`javascript:alert(1)`
- 复扫结果：
  - 失效 `/uploads` 引用：`0`
  - 非法媒体 URL：`0`

## 工具说明
```bash
# 只读巡检
python repair_upload_references.py --json

# 执行修复
python repair_upload_references.py --execute --json
```

脚本策略：
- 对 `backend_users/technicians/promotions` 的坏引用执行 `set NULL`
- 对 `store_images/store_portfolio` 的坏引用删除行
- 对 `reviews.images` 里的坏图片做剔除
- 对 `pins.image_url`：
  - 已删除 pin 且图片失效：自动删除
  - 未删除 pin 且图片失效：停止并要求人工处理
