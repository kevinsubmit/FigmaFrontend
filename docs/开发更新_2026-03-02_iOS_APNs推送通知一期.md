# 开发更新（2026-03-02）：iOS APNs 推送通知一期

## 本次目标
- 为 iOS App 接入 APNs 推送能力。
- 支持设备 token 注册/注销。
- 后端创建站内通知时，同步尝试发送 APNs 推送。
- iOS 端点击系统通知后跳转到 `Appointments` 页面。

## 后端改动
- 新增表：`push_device_tokens`
  - 字段包含：`user_id`、`platform`、`device_token`、`apns_environment`、`is_active`、`last_seen_at` 等。
- 新增 CRUD：设备 token 的 upsert、注销、按用户查询、失效批量停用。
- 新增 API（登录态）：
  - `POST /api/v1/notifications/devices/register`
  - `POST /api/v1/notifications/devices/unregister`
- 新增 APNs 服务：
  - 使用 `.p8` 私钥 + JWT（ES256）调用 APNs。
  - 支持 `sandbox/production` 环境路由。
  - APNs 返回失效 token 原因时自动停用对应 token。
- 通知联动：
  - `notification_service.create_notification(...)` 在写入通知后尝试推送（失败不影响主流程）。
  - reminder 流程改为统一走 `notification_service`，确保可以触发推送。

## iOS 改动
- App 启动后配置 `UNUserNotificationCenter` delegate。
- 登录后自动：
  - 请求通知权限（首次）。
  - 注册远程通知并上报 APNs token 到后端。
- 登出时自动注销设备 token。
- 跳转策略：
  - 前台收到推送：仅展示系统通知（banner/list/sound）。
  - 用户点击通知：跳转到 `Appointments`。

## 配置项（后端）
在 `backend/.env` 中新增/使用：
- `APNS_ENABLED`
- `APNS_KEY_ID`
- `APNS_TEAM_ID`
- `APNS_BUNDLE_ID`
- `APNS_PRIVATE_KEY` 或 `APNS_PRIVATE_KEY_PATH`
- `APNS_USE_SANDBOX`
- `APNS_TIMEOUT_SECONDS`

## 联调建议
1. 先跑迁移，确认 `push_device_tokens` 表存在。
2. 后端开启 APNs 配置并重启服务。
3. iOS 真机开启 `Push Notifications` 能力并安装调试包。
4. 登录后检查后端是否收到 `/notifications/devices/register`。
5. 触发一条通知（例如预约提醒）并验证真机是否收到。
6. 点击通知后应进入 `Appointments` 页面。
